## What it does

You cloned a repo. `codeglance` answers the five questions you'd otherwise spend 15 minutes on:

- **What is this?** — "Next.js 14 with Prisma, tRPC" not just "TypeScript"
- **How do I run it?** — real commands extracted from `package.json`, `Makefile`, `Cargo.toml`, `go.mod`
- **Where do I start?** — entry points + key files ranked by importance
- **What tools does it use?** — CI, test runner, Docker, linting, env files
- **How do I brief an LLM?** — `--for-ai` generates a compact, structured context brief

Works best on Node.js, Python, Go, Rust, C/C++, and mixed repos.
Falls back gracefully on anything else — it reads what it can from manifest files and structure.

## Usage

```bash
codeglance [path]                   # analyze current dir or given path
codeglance --markdown               # Markdown report (for docs/)
codeglance --for-ai                 # compact LLM context brief
codeglance --json                   # machine-readable output
codeglance --output codebase-tour.md  # save to file
codeglance --no-git                 # skip git analysis
codeglance --help                   # all options
```

### Generate a living onboarding doc

```bash
codeglance --markdown --output docs/codebase-tour.md
```

Check this file into your repo. Regenerate when the architecture changes. Your team will thank you.

### Brief an LLM

```bash
codeglance --for-ai | pbcopy    # macOS — paste into Claude, GPT, Gemini
codeglance --for-ai | xclip     # Linux
```

You get a compact, structured brief (≈400 tokens) with stack, commands, entry points, and tooling.
No source code dumped. No token waste.

### Use in CI

```bash
# GitHub Actions:
- run: npx codeglance --json > codebase-report.json
```

## Why not tokei / scc / repomix / code2prompt?

These are all good tools. Use the right one for the job:

- **tokei / scc** — use these when you want fast, accurate line-of-code counts.
- **repomix / code2prompt** — use these when you want to pack source code *for* LLM consumption.
- **codeglance** — use this when you want to *understand* a repo before you write a line.

| | codeglance | tokei/scc | repomix |
|---|:---:|:---:|:---:|
| Framework detection | ✓ | ✗ | ✗ |
| Run/build/test commands | ✓ | ✗ | ✗ |
| Entry points | ✓ | ✗ | ✗ |
| "Files to read first" | ✓ | ✗ | ✗ |
| CI / Docker / tooling | ✓ | ✗ | ✗ |
| Language stats | ✓ | ✓ | ✗ |
| LLM context brief | ✓ | ✗ | ✓ (full source) |
| Zero config | ✓ | ✓ | ✓ |

## Limitations

Be clear-eyed about what codeglance is:

- **Heuristic, not semantic.** It reads manifest files and file structure. It does not parse or understand source code.
- **Framework detection depends on manifests.** If a project doesn't use a standard package file, detection may be shallow.
- **"Start here" ranking is heuristic.** It uses file depth, naming patterns, and size — not import analysis. Results are approximate.
- **Large repos are capped.** Repos over 25,000 files are scanned up to that cap. The output includes a notice.
- **Git requires a local git binary.** If git isn't installed or the directory isn't a repo, the git section is silently omitted.
- **Unknown ecosystems fall back gracefully.** You'll still get language stats and file structure, but no framework or command info.

codeglance says "heuristic" in its output. It does not pretend to be smarter than it is.

## How it works

codeglance never reads source code. It reads manifest files and checks for the presence of known files.

- `package.json`, `go.mod`, `Cargo.toml`, `pyproject.toml`, `CMakeLists.txt` → framework names + versions
- `scripts` in `package.json`, `Makefile` targets, `cargo`/`go`/`cmake` conventions → commands
- Presence of known files and directories → entry points, CI, Docker, test runner
- Heuristic scoring of source files → "files to read first"
- Local `git` CLI → branch, last commit, recent activity

Everything is transparent. When something is a heuristic, the output says so.

## Contributing

Each ecosystem is a self-contained detector module. **Adding a framework is ~3 lines of code.**

```typescript
// src/detectors/frameworks.ts — add to the relevant array:
const NODE_FRAMEWORKS: FrameworkDef[] = [
  // ...
  { name: 'My Framework', category: 'web_framework', keys: ['my-framework-package'] },
];
```

Add a fixture + test and submit a PR. See [CONTRIBUTING.md](CONTRIBUTING.md) for the full guide.

**Good first issues:**
- Add Laravel / Symfony detector
- Add Phoenix / Elixir detector
- Add Spring Boot / Quarkus improvements
- Add pnpm workspace / monorepo detection
- Improve C++ Conan/vcpkg support

Open issues labeled [`good first issue`](https://github.com/nicholasess/codeglance/issues?q=label%3A%22good+first+issue%22).
